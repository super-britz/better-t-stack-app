# IAM 最佳实践

## 核心概念

| 概念 | 说明 |
|------|------|
| User | 代表一个人，有账号密码和 Access Key |
| Role | 代表一个服务，AWS 自动颁发临时凭证 |
| Policy | 权限规则，定义能做什么，可以附加给 User 或 Role |
| Group | 用户组，把 Policy 附加给组，人加入组继承权限 |

Policy 是规则，User 和 Role 是规则的载体，载体是人还是服务无所谓。

---

## 最佳实践

### 1. 根用户只做两件事，平时不用

```
开启 MFA → 创建第一个 IAM 管理员用户 → 退出登录 → 锁起来
```

根用户拥有账号所有权限，包括删除账号、修改账单，一旦泄露后果是灾难性的。只在以下情况才用：
- 账单相关操作
- IAM 管理员出问题需要救场

---

### 2. 给根用户和管理员都开 MFA

```
根用户        →  必须开
管理员 IAM User →  必须开
```

即使密码泄露，没有手机也登不进去。

---

### 3. 每个人一个 IAM User，不共用

```
张三一个 User，李四一个 User
出问题可以精确定位是谁操作的
```

共用账号出了问题无法追责，也无法精确吊销某一个人的权限。

---

### 4. 用组管理权限，不给个人单独附加 Policy

```
创建"开发组" → 附加 Policy → 把人加入组
而不是给每个人单独附加 Policy
```

人员变动时只需要把人从组里加/移除，不需要逐个调整权限。

---

### 5. 服务用 Role，不用 User

```
Lambda、EC2 等服务  →  挂 Role，AWS 自动颁发临时凭证
不要把 Access Key 写进代码或环境变量
```

Role 的临时凭证几小时自动过期，泄露了影响有限。Access Key 长期有效，泄露了就一直有效。

---

### 6. 最小权限原则

```
只给够用的权限，不给多余的
Lambda 不需要 S3 就不给 S3 权限
```

即使服务被攻击或代码有漏洞，攻击者拿到的权限也非常有限，做不了什么破坏。

---

### 7. 定期轮换 Access Key

```
Access Key 长期不换，泄露了都不知道
定期换一次，旧的作废
```

---

## 权限分配流程

```
根用户（权限起点，拥有一切）
  ↓ 创建 IAM 管理员 User
管理员
  ↓ 创建 Policy（权限规则）
  ↓
  ├── 附加给 User/Group  →  人
  └── 附加给 Role        →  服务（Lambda、EC2 等）
```

---

## 人 vs 服务 对比

| | 人（User） | 服务（Role） |
|--|-----------|------------|
| 凭证 | Access Key，长期有效 | 临时凭证，几小时过期 |
| 获取方式 | 手动创建，自己保存 | AWS 自动颁发，自动轮换 |
| 泄露风险 | 高 | 低 |
| 适用场景 | 人操作 CLI / 控制台 | Lambda、EC2 等服务 |
